# PLUXX-317 Review

## Scope

Review of the field-level translation registry, YAML frontmatter parsing and rewriting, generated compatibility truth, and their CLI/generator consumers for PLUXX-317.

## CE findings resolved

- Required a frontmatter closing fence at column zero so indented `---` content inside block scalars is not mistaken for the document boundary.
- Replaced nested-key-sensitive line mutation with a shared top-level field rewrite helper, including complete removal of block sequence children.
- Converted YAML aliases and unsupported tags into located diagnostics instead of allowing parser conversion to throw or silently ignoring warnings.
- Reclassified compatibility-only skill metadata from `translate` to `degrade`; no host is credited with native controls that its generator does not produce.
- Defined mixed field loss as `degrade`; `drop` is reserved for primitives where every audited field is dropped.
- Reused parsed frontmatter and shared summary helpers to keep lint, migration, generators, validation, and documentation on the same truth path.

## Specialized reliability and supply-chain review

- `yaml` is pinned exactly at `2.9.0`; the lockfile contains its registry integrity hash and the package supports the repository's Node 18 floor.
- The runtime build externalizes `yaml`, matching the package's declared runtime dependency and avoiding unsupported bundled dynamic loading.
- No new package-specific finding was identified by the independent review.
- A fresh live `npm audit` could not be authorized during this work block. This is a proof limitation, not evidence that the dependency tree is vulnerability-free.

## Validation evidence

- Targeted parser, lint, migration, generator, hook, and platform-rule checks passed after review fixes.
