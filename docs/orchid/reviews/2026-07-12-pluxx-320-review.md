# PLUXX-320 Release Integrity Review

## Scope

Reviewed the PLUXX-320 diff against `origin/main` for release integrity, staged recovery, resumable npm/GitHub publication, and upgrade provenance. The review covered generated installers, archive validation, publish orchestration, the tag release workflow, CLI upgrade behavior, tests, and public planning truth.

## Review outcome

The CE correctness/reliability review and Codex Security supply-chain review found actionable gaps during implementation. All were addressed before closeout:

- custom release base URLs now propagate through the top-level installer, and `install-all.sh` delegates through the verified front door
- installer staging occurs beside the destination, with owned companion metadata included in rollback
- downloaded installers and archives are checksum-verified before execution or extraction; absolute paths, traversal, symlinks, and hard links fail closed
- signal exits preserve failure status and trigger rollback
- source, requested, built, and package versions must all exist and agree
- npm reconciliation compares exact packed-tarball SRI; the workflow publishes the same verified tarball bytes
- GitHub reconciliation verifies the exact expected asset set and removes stale extras
- upgrades accept only `latest` or exact semver, inspect the active CLI package without executing a PATH-selected binary, and report direction plus rollback instructions

The sealed diff scan reported zero surviving findings after remediation. Canonical scan artifacts remain in the local Codex security artifact store and are intentionally not committed to the public repo.

A subsequent Blocks review found two remaining name-only GitHub asset checks; both now download remote assets and compare their bytes to the local SHA-256 or SHA-512 SRI, with the expanded 605/605 suite passing afterward.

## Residual risk

- No live npm or GitHub release mutation was performed; remote reconciliation is covered by focused tests and must also pass CI/release execution.
- The existing development dependency audit reports three toolchain-only advisories (low `esbuild`, high `vite`, critical `vitest`); production dependencies remain clean. Dependency upgrades are outside PLUXX-320 and should be handled separately.

## Main integration proof

After seven sibling audit PRs landed, `origin/main` at `5a06983` was merged into this branch without rebasing or force-pushing. Conflict resolution preserved current main's proof freshness, atomic authoring mutations, MCP tape safety, semantic eval, translation registry, safe docs ingestion, and ownership-aware install transactions while retaining PLUXX-320's release checksum/manifest enforcement, exact remote-byte verification, hardened upgrade path, signal exits, and rollback of owned Claude/Codex companion state.

Post-integration validation:

- focused publish/recovery suite: 38 passed
- focused release workflow, upgrade, install ownership, filesystem transaction, proof freshness, package, meta-CLI, and distribution lifecycle suites passed
- `npm run typecheck` and `npm run build` passed
- official serialized `npm test`: 59 files, 702 tests passed
- packed-package runtime verification and package dry-run passed

The post-merge CE review then exercised the accumulated-main composition rather than relying on the pre-merge audit. It found and fixed additional integration failure modes:

- generated host installers pin their own tagged release instead of resolving a moving `latest`
- live-directory swaps recover when interrupted between the backup and stage moves, and an install-scoped lock prevents concurrent installers from racing
- the bundle ownership record is backed up and replaced atomically with the installed bundle
- OpenCode bundle and companion state reach one commit point before backups become disposable
- remote downloads use bounded retries/timeouts, upgrade subprocesses have a finite timeout, and ambiguous npm/GitHub lookups fail closed
- successful publish results always require exact remote verification; the public verification bypass was removed
- prerelease SemVer ordering, non-mutating upgrade help, install-failure rollback output, and agent-facing upgrade recovery guidance are covered
- release-workflow tests parse the YAML step order, and the workflow no longer treats npm transport/auth failures as an unpublished version

The maintainability reviewer also recommended splitting the already-large publish module. That refactor is intentionally not part of PLUXX-320: it does not change the release-integrity acceptance behavior and would expand the conflict surface immediately after the audit-PR integration. It remains a focused follow-up opportunity rather than a release blocker.
