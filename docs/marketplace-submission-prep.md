# Marketplace Submission Prep For `pluxx publish` Flows

This document defines the docs-ready deliverable for marketplace-aware publish flows.

It covers:

- Claude Code submission expectations
- Cursor marketplace and team marketplace submission prep
- what Pluxx can generate automatically vs what remains manual at submission time

It does not implement `pluxx publish` yet.

## Docs-Ready Checklist

Use this checklist before we call publish flow work complete.

- [ ] `pluxx doctor` passes with no blocking errors
- [ ] `pluxx test` passes for target bundles
- [ ] `dist/claude-code/` and `dist/cursor/` are freshly generated from current source
- [ ] generated manifests are reviewed:
  - `dist/claude-code/.claude-plugin/plugin.json`
  - `dist/cursor/.cursor-plugin/plugin.json`
- [ ] MCP config output is reviewed:
  - `dist/claude-code/.mcp.json`
  - `dist/cursor/mcp.json`
- [ ] hooks output is reviewed when hooks are enabled:
  - `dist/claude-code/hooks/hooks.json`
  - `dist/cursor/hooks/hooks.json`
- [ ] instructions/rules output is reviewed:
  - `dist/claude-code/CLAUDE.md`
  - `dist/cursor/AGENTS.md`
  - `dist/cursor/rules/*.mdc` (if cursor rules are configured)
- [ ] all manual submission copy/assets are prepared (see "Manual Inputs" below)

## Platform Expectations

### Claude Code

Current expectation: Claude Code has a formal plugin marketplace surface (`.claude-plugin/marketplace.json`) documented in this repo's extension-systems research. For publish v1, Claude should be treated as a marketplace-capable plugin target, with local marketplace registration still useful for testing and internal rollout.

Submission prep for Claude therefore means:

- bundle quality gates pass (`doctor`, `lint`, `test`)
- manifest and plugin content are complete for distribution
- marketplace metadata is complete and coherent for the intended Claude marketplace flow
- local marketplace install path remains valid for testing and internal rollout

### Cursor

Cursor supports marketplace and team marketplace distribution with manual review in the current research docs.

Submission prep for Cursor means:

- `.cursor-plugin/plugin.json` is complete and coherent
- plugin assets and listing copy are ready for manual review workflows
- generated rules/hooks/MCP references point to valid in-bundle files

|---|---|---|---|---|---|
| Plugin name | `.claude-plugin/plugin.json:name` | `.cursor-plugin/plugin.json:name` | `name` | Yes | No |
| Description | `.claude-plugin/plugin.json:description` | `.cursor-plugin/plugin.json:description` | `description` (+ optional `brand.shortDescription` for docs content) | Yes | No |
| Version | `.claude-plugin/plugin.json:version` | `.cursor-plugin/plugin.json:version` | `version` | Yes | No |
| Author | `.claude-plugin/plugin.json:author` | `.cursor-plugin/plugin.json:author` | `author` | Yes | No |
| License | `.claude-plugin/plugin.json:license` | `.cursor-plugin/plugin.json:license` | `license` | Yes | Usually no |
| Repository URL | `.claude-plugin/plugin.json:repository` | `.cursor-plugin/plugin.json:repository` | `repository` | Yes (if set) | Usually no |
| Keywords | `.claude-plugin/plugin.json:keywords` | `.cursor-plugin/plugin.json:keywords` | `keywords` | Yes (if set) | Usually no |
| Homepage | N/A in current generator | `.cursor-plugin/plugin.json:homepage` | `brand.websiteURL` | Yes for Cursor (if set) | Possibly |
| Logo/Icon URL | N/A in current generator | `.cursor-plugin/plugin.json:logo` | `brand.icon` | Yes for Cursor (if set) | Often |

| Compliance/policy attestations | external | external | external | No | Yes |

## Manual Inputs

These items are expected to stay manual even after publish v1 is introduced:

- marketplace listing headline and long-form description
- submission media pack (screenshots, demo video, logo variants)
- support and policy links when required by host submission UI
- release notes and reviewer-facing test instructions
- final policy and security attestations in host submission forms

## Suggested `pluxx publish` v1 Docs Contract

For docs and CLI messaging, publish v1 should make this split explicit:

- `Generated`: manifests, MCP config, hooks, rules/instructions, platform bundle structure
- `Manual`: listing copy/media/compliance fields that are host-side and not fully file-backed

That keeps publish flows honest while still making the workflow fast.
